# ODISS 서버엔진 패치노트

> 작성일: 2026-04-09
> 목적: 이후 작업자(모델)가 현재 상태를 빠르게 파악하고 승계할 수 있도록 작성

---

## 1. 프로젝트 개요

**ODISS (OCR-based Drug Information Smart Speaker)** — 시니어 복약지도 AI 서버엔진.
로컬 에이전트(스마트 스피커)와 WebSocket으로 통신하는 클라우드 서버.

아키텍처 원본: `server.mermaid` (Mermaid flowchart)

---

## 2. 이전 상태 (Before)

- `app/services/smart_speaker.py`: 로컬 마이크/스피커를 직접 제어하는 모놀리식 스크립트 (PyTorch, Whisper, Qwen TTS, GLM-OCR 모두 한 파일)
- FastAPI 앱은 존재했으나 OCR → DUR → LLM 단방향 파이프라인만 구현
- **SQLAlchemy** 기반 SQLite DB 사용 (`app/models/db.py`)
- 엔진 분리 없음, 추론/메모리/대화 로직이 혼재

---

## 3. 수행한 작업 요약

### 3-1. 아키텍처 재설계 (server.mermaid 기반)

`smart_speaker.py`의 로컬 제어 방식을 **FastAPI 기반 완전한 API 서버**로 전환.
4대 엔진 + Tool Execution Engine 구조로 분리.

### 3-2. SQL 완전 제거 → MD 파일시스템 데이터베이스

- SQLAlchemy, aiosqlite, pymysql 등 **SQL 의존성 전부 삭제**
- `app/models/db.py` 비활성화 (빈 파일)
- `requirements.txt`에서 DB 관련 패키지 모두 제거
- 모든 데이터를 **날짜별 디렉토리 + 개별 Markdown 파일**로 저장

### 3-3. Docker 환경 구성

- `Dockerfile` + `docker-compose.yml` 신규 생성
- `docker compose up --build` 로 실행 가능

### 3-4. 대화엔진 최종 검토/로컬 발화 파이프라인 추가 (2026-04-29)

사용자에게 전달되는 복약 답변의 안전성과 자연스러운 발화를 분리하기 위해 `/ws/chat` 대화 흐름을 다음 구조로 확장했다.

```
STT 입력
 → ConversationEngine 입력/대기 멘트
 → ReasoningEngine Tool/DUR 기반 핵심 팩트 생성
 → GPT Judge 최종 안전 검토
 → Local LLM 최종 사용자 발화 생성
 → WebSocket response 전송
```

변경 이유:

- 추론엔진은 DUR/HIRA/건기식 tool 실행과 핵심 팩트 생성에 집중한다.
- GPT 계열 Judge는 사용자에게 전달되기 전의 위험한 단정, 처방 변경 지시, 임의 중단/병용 표현을 최종 점검한다.
- 로컬 모델은 검토된 문장을 고령 사용자가 듣기 쉬운 짧은 답변으로 바꾼다.
- 외부 GPT 검토와 로컬 발화 생성을 분리해, 안전성 검토와 사용자 경험을 각각 독립적으로 개선할 수 있게 한다.

주요 구현:

- `app/engines/llm_judge.py`
  - `review_final_answer()` 추가
  - `OPENAI_JUDGE_MODEL` 설정을 우선 사용하며 기본값은 `gpt-5`
  - OpenAI 키가 없으면 핵심 팩트를 그대로 반환하는 안전 fallback 제공
- `app/services/llm.py`
  - `call_local_delivery_llm()` 추가
  - GPT Judge 검토문을 로컬 LLM이 최종 발화 문장으로 변환
  - 로컬 LLM 서버가 없으면 검토문에 안전 문구를 붙여 반환
  - 기존 내부 LLM mock 응답이 특정 약효를 추측하지 않도록 안전 fallback으로 수정
- `app/api/routes/agent_ws.py`
  - `_handle_stt()` 흐름을 `core_message → GPT Judge review → local delivery → response`로 변경
  - Judge 검토용으로 복약 요약, 관련 메모리, DUR/건기식/HIRA 결과를 `_build_review_context()`에서 최소 맥락으로 구성
- `app/prompts/llm_prompts.json`, `app/services/prompt_registry.py`
  - `judge_final_review`, `local_delivery` 프롬프트 추가
- `app/core/config.py`, `.env.example`
  - `INTERNAL_LLM_MODEL`
  - `OPENAI_JUDGE_MODEL`
- `tests/test_conversation_pipeline.py`
  - GPT 키/로컬 LLM 서버가 없는 상황에서도 안전 fallback이 동작하는지 검증

### 3-5. ODISS 데모 대화, 성능, UX 개선 (2026-05-13)

ODISS 데모 시나리오를 자연스럽게 통과하고, 실제 사용자가 기다리는 시간이 길어지지 않도록 대화엔진과 추론 경로를 함께 개선했다.

핵심 목표:

- OCR 저장, DUR 확인, 복약 계획처럼 오래 걸리던 경로를 빠르게 만든다.
- 시니어뿐 아니라 중년 만성질환자도 자연스럽게 사용할 수 있는 말투와 호칭으로 바꾼다.
- 최종 사용자가 듣는 답변은 대화엔진이 책임지므로, 프롬프트와 톤 정책을 ODISS UX 기준으로 정리한다.
- 오래 걸릴 수 있는 작업은 사용자가 불안하거나 지루하지 않도록 즉시 filler 멘트를 보낸다.

#### 3-5-1. 병목 원인 분석

측정된 10초 이상 대표 병목:

| 케이스 | 기존 지연 | 주요 원인 |
|--------|-----------|-----------|
| OCR store | 약 83초 | OCR 저장 시 DUR 동기화를 끝까지 기다림 |
| DUR all categories | 약 62초 | 모든 약에 대해 T2~T10 전체 DUR 호출 |
| simple daily medication plan | 약 49초 | 메모리 조회 + DUR + LLM 후처리가 한 흐름에서 대기 |
| seeded query ask | 약 34초 | frontier LLM fallback / 최종 검토 / delivery LLM 대기 가능성 |
| OCR meds plan / DUR check | 약 25~28초 | 다약제 DUR 순차 호출 및 긴 timeout |

가장 큰 문제는 "모든 약 x 모든 DUR API" 방식이었다. 예를 들어 약 4개를 T2~T10 전체로 확인하면 `4 x 9 = 36`번의 외부 API 호출이 발생한다. 따라서 질문 의도에 필요한 DUR만 호출하도록 구조를 바꿨다.

#### 3-5-3. OCR 저장 흐름 개선

변경 파일:

- `app/api/routes/agent_ws.py`
- `app/api/routes/ocr_api.py`

기존 흐름:

```
OCR 결과 수신
 → OCR 기록 저장
 → 모든 약 DUR 확인 대기
 → 처방 메모리 저장
 → 사용자에게 완료 응답
```

개선 후 흐름:

```
OCR 결과 수신
 → 흐리거나 약 이름 부족하면 저장하지 않고 재촬영 안내
 → 충분하면 "이 정보를 저장할까요?" 확인 요청
 → 사용자가 "응, 저장해줘"라고 하면 즉시 복약 정보 저장
 → 사용자에게 바로 저장 완료 응답
 → DUR 동기화는 background task로 진행
```

효과:

- 사용자는 DUR 완료까지 기다리지 않는다.
- OCR 결과가 불확실하면 추측 저장하지 않는다.
- 저장 확인 단계가 생겨 데모 대화 흐름이 자연스러워졌다.

#### 3-5-7. 기다리는 동안의 filler UX 개선

기존 문제:

- `ConversationEngine.generate_filler()`는 있었지만 WebSocket에서는 긴 작업이 끝난 뒤 전송되는 경우가 있어, 실제 latency hiding 효과가 약했다.

개선 후:

- WebSocket에서 긴 작업을 시작하기 전에 filler를 먼저 전송한다.
- 발화 내용에 따라 filler 문구를 다르게 선택한다.

상황별 예시:

| 상황 | filler 예시 |
|------|-------------|
| OCR | "사진에서 읽힌 약 이름을 확인하고 있어요." |
| DUR/상호작용 | "약끼리 같이 드셔도 되는지 안전 정보를 확인하고 있어요." |
| 복약 기록 | "방금 드셨다는 내용을 기록에 맞게 정리하고 있어요." |
| 알림 설정 | "아침, 점심, 저녁 알림 시간을 맞춰보고 있어요." |
| 일반 복약 질문 | "복용 중인 약과 관련 기록을 함께 확인하고 있어요." |

효과:

- 사용자가 기다리는 이유를 알 수 있다.
- 의료 판단 대기 중에도 불안감을 줄인다.
- "잠시만요"만 반복되는 느낌을 줄였다.

#### 3-5-8. 프롬프트 개선

변경 파일:

- `app/prompts/llm_prompts.json`
- `app/services/prompt_registry.py`
- `app/prompts/qwen35_reasoning_template.md`

공통 원칙:

- 결론 먼저, 근거는 짧게, 다음 행동은 명확하게 말한다.
- 위험은 숨기지 않되 겁주지 않는다.
- 확인 가능한 정보와 불확실한 정보를 구분한다.
- 사용자가 바로 할 수 있는 행동은 1~2개만 제안한다.
- `<think>`나 내부 추론은 출력하지 않는다.
- 모든 DUR을 습관적으로 호출하지 말고 필요한 DUR만 선택한다.

`main_answer`:

- ODISS 역할을 "복약 상담 보조 + 복약 관리 대화 행위 수행"으로 명확화
- 신규 등록, 프로필 회상, OCR 촬영/재촬영, 저장 확인, 식후 복약 안내, 알림 설정, 복용 기록을 ODISS 대화 행위로 명시

`judge_final_review`:

- 의료 안전 판단이 필요한 경우만 상담 문구를 유지/추가
- 비의료 대화 행위는 원래 의도를 보존
- 최종 문장은 사용자에게 들려줄 수 있는 형태로 정리

`local_delivery`:

- 이름 기반 호칭 사용
- 1~4문장으로 짧게 말하기
- 약 이름, 시간, 횟수를 또렷하게 말하기
- 검토문에 없는 의학 정보를 새로 만들지 않기
- 위험 표현은 유지하되 겁주지 않는 톤 사용


## 5. MD 파일시스템 데이터베이스 (`app/database/md_store.py`)

### 설계 원칙

- 단일 MD 파일에 append 하지 않음
- **매 기록이 독립된 .md 파일**로 저장됨
- 날짜별 디렉토리로 자동 분류

### 런타임 디렉토리 구조 (data/md_database/)

```
permanent/
├── patients/{user_id}/profile.md        사용자별 프로필
├── ocr_history/
│   ├── 2026-04-09/
│   │   ├── 001_153022.md                시간순 일련번호
│   │   └── 002_164510.md
│   └── 2026-04-10/
│       └── 001_091500.md
├── prescriptions/YYYY-MM-DD/NNN.md      처방전 + DUR 동기화 결과
├── medication_log/YYYY-MM-DD/NNN.md     상담 질문/응답 기록
├── dur_linkage/YYYY-MM-DD/NNN.md        DUR API 호출 기록
└── health_supplement/YYYY-MM-DD/NNN.md  건기식 조회 기록
flash/
├── prescription_log.md                  현재 복용 요약 (덮어쓰기)
├── current_user_profile.md              현재 대화자 (덮어쓰기)
├── current_requirement.md               최근 5건 요구사항 (덮어쓰기)
├── current_manual.md                    응답 매뉴얼 (덮어쓰기)
└── context_memory.md                    대화 컨텍스트 (덮어쓰기)
```

### 주요 API

| 메서드 | 설명 |
|--------|------|
| `store.save(category, content)` | 날짜 디렉토리에 새 MD 파일 생성 |
| `store.search(category, keyword, start=, end=)` | 날짜 범위 + 키워드 탐색 |
| `store.list_entries(category, start=, end=)` | 날짜 범위로 파일 목록 |
| `store.read_latest(category, n=5)` | 최신 N개 엔트리 읽기 |
| `store.read_entry(path)` | 단일 파일 읽기 |
| `store.save_user_file(user_id, filename, content)` | 사용자별 파일 저장 |
| `store.read_user_file(user_id, filename)` | 사용자별 파일 읽기 |
| `store.user_exists(user_id)` | 사용자 등록 여부 |
| `store.read_flash(key)` / `store.write_flash(key, content)` | Flash 메모리 R/W |

### 구조화 메모리 (`app/memory/`)

- `app/memory/`는 Claude Code 방식처럼 `data/md_database/structured_memory` 아래에 별도 장기 메모리를 둔다.
- 저장 구조는 `global/`과 `speakers/{speaker_id}/`로 나뉘며, 각 디렉터리는 `MEMORY.md` 인덱스와 topic `.md` 파일을 가진다.
- topic 파일은 frontmatter의 `name`, `description`, `type`을 사용하고, `type`은 `user`, `feedback`, `project`, `reference` 중 하나다.
- `MEMORY.md`에는 본문을 넣지 않고 topic 파일 링크와 한 줄 설명만 둔다.
- `scan.py`는 topic 파일 앞부분만 읽어 메모리 헤더를 만들고, `selector.py`는 질의와 매칭되는 topic 파일만 relevant memory로 고른다.
- `engines/memory.py`는 기존 `flash/permanent` 기록을 유지하면서, 환자 프로필과 최신 OCR+DUR 복약 맥락을 `structured_memory`에도 동기화한다.

## 8. WebSocket 데이터 흐름 (`/ws/chat`)

### 클라이언트 → 서버 (수신)

```json
{ "type": "stt_result", "text": "이 약 먹어도 돼?", "speaker_id": "user_001" }
{ "type": "ocr_result", "data": { "medications": [...] }, "speaker_id": "user_001" }
{ "type": "ping" }
```

### 서버 → 클라이언트 (송신)

```json
{ "type": "filler", "text": "잠시만요, 기록을 확인하고 있어요." }
{ "type": "response", "response_text": "...", "response_type": "medical_response", "requires_tts": true }
{ "type": "ocr_request", "action": "request_ocr", "message": "처방전 사진을 찍어주세요." }
{ "type": "ocr_processed", "message": "3개 약품의 DUR 확인이 완료되었습니다." }
{ "type": "pong" }
{ "type": "error", "message": "..." }
```

### 내부 파이프라인 순서 (agent_ws.py `_handle_stt`)

```
1. CE_Input       → conversation_engine.receive_input()
2. CE_Latency     → conversation_engine.generate_filler() → WebSocket 즉시 전송
3. 스몰토크 체크   → 스몰토크이면 바로 응답 반환
4. ME_Context     → memory_engine.load_context()
5. RE_Intent      → reasoning_engine.classify_intent() + plan_tasks()
6. Tool 실행      → reasoning_engine.execute_tasks() (DUR/HIRA/RAG/보충제)
7. RE_Core_Msg    → reasoning_engine.synthesize_core_message(verify_with_judge=False)
8. GPT Judge      → llm_judge.review_final_answer()로 최종 안전 검토
9. Local LLM      → call_local_delivery_llm()로 사용자 발화 생성
10. CE_Response   → conversation_engine.build_response() → WebSocket 전송
11. ME_Update     → memory_engine.update_and_compress() → MD 파일 저장
```

현재 WebSocket 대화 경로에서는 `ReasoningEngine`이 핵심 팩트만 만들고, 최종 검토는 `LLMJudgeEngine.review_final_answer()`가 맡는다. 이후 `call_local_delivery_llm()`이 검토된 문장을 로컬 모델에 전달해 실제 사용자에게 들려줄 답변으로 변환한다. 로컬 LLM 서버가 없으면 검토문을 안전 문구와 함께 그대로 사용한다.

## 11. 알려진 한계 및 남은 작업

### 미구현 / 미완성

1. **의도 분류(RE_Intent)가 키워드 기반** — LLM 기반 intent classification 으로 고도화 필요
2. **ME_RAG 검색이 단순 키워드 매칭** — 임베딩 기반 시맨틱 검색이나 BM25 도입 필요
3. **다중 대화자 식별(Speaker Diarization)** — 현재 `speaker_id`를 클라이언트가 직접 전달해야 함. 음성 특징 기반 자동 식별은 미구현
4. **우선순위 큐(Priority Queue)** — 응급 상황 감지 시 즉시 경고하는 로직은 코드에 있으나, 대화 중간 끊기(interrupt) 메커니즘은 미구현
5. **대화형 버퍼** — 메모리 엔진 처리 중 추가 질문이 들어올 때 큐잉하는 로직 미구현
6. **로컬 에이전트(클라이언트) 코드** — 서버만 구현됨. STT/TTS를 수행하는 로컬 에이전트 클라이언트는 별도 구현 필요
7. **로컬 LLM 서버 연결** — 서버 코드는 OpenAI-compatible `INTERNAL_LLM_API_URL`을 호출하도록 준비됨. 실제 Qwen/EXAONE/vLLM 서버 기동 및 모델명 설정은 배포 환경에서 필요
8. **GPT Judge 실제 모델 가용성** — 기본 설정은 `OPENAI_JUDGE_MODEL=gpt-5`이지만, 사용하는 OpenAI 계정/엔드포인트에서 해당 모델명이 지원되는지 확인 필요

### 기존 코드 중 미정리
